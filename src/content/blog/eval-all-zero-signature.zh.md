上一版测试集跑起来了，但每个 case 都判 0 分，整套评估体系随之被废弃，团队从此没有评估。复盘定性：系统天天在用，不可能全错——全 0 分说明评分器从头到尾没匹配上，问题在「期望值怎么写、怎么打分」，不在被测系统。

## 可复用的判断框架

- 多 Agent 系统的期望工具轨迹必须包含路由转交（transfer）、嵌套包装、内部工具——按单 Agent 写期望值，精确序列匹配下必然全 0。
- 有「暂停等确认」环节的流程，评估脚本必须把确认喂回去，否则所有 case 卡死在确认步。
- 自由文本输出不能用精确字符串匹配打分，要在结构化交接点打分。
- 评估基建的铁律：单元 / 集成层 Mock 零依赖、一条命令可跑、进 CI、有所有权——上一版死因就是这四样全缺。

> 最快诊断手法：任取一个 case，实际轨迹 vs 期望值并排 diff，根因通常第一行就暴露。

并排一看就清楚——期望值漏了路由转交那一步，于是整条精确序列匹配从第一步就错位，全程判 0：

```text
expected: [search, answer]
actual:   [transfer_to_specialist, search, answer]
                                    ^ 期望值漏了这一步 → 序列错位 → score 0
```

把期望轨迹补上转交节点、改成在交接点打分，分数立刻恢复正常。
