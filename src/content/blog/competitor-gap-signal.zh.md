在某 B 端商家 AI Agent 项目立项时，竞品调研发现全行业几乎只有一家成熟 SaaS 做了「写操作」。当时把它解读为「写操作才是 Agent 的核心，这是机会」，沿着这个判断投入了两个多月开发，最后才被证实不可行——写原语没有被服务化，协作框架也撑不住。这不是廉价试错，是季度级的方向性沉没成本，这正是这条教训值钱的原因。

## 方法论

- 「竞品都没做」是双义信号——可能是机会，也可能是大家都撞了墙。看到它，下一个问题必须是「他们是没想到，还是做不到」。某些公司能做成不是因为「拥有全栈」（谁都拥有），而是它的生态让对外 API 被外部开发者逼着打磨了很多年；平台型公司的写逻辑往往长在各业务团队的 UI 流程里，「公司拥有全栈」≠「我的团队能用」（Conway's law）。
- **结论不可前置，但验证可以前移。** 立项前花几小时摸三个高信号问题：这个 API 现在的消费者是谁（只有自家前端在调 → 大概率是 BFF 接口，Agent 接上去处处别扭）；有没有 dry-run / preview 能力；权限模型按操作粒度还是页面粒度。这三问不能告诉你「行」，但经常能提前告诉你「悬」。
- 剩下消不掉的不确定性用顺序解决：把链路里风险最高的那个写调用剥出来先做端到端 spike，先于一切功能开发。同样会撞墙，但撞在第一周，而不是两个多月之后。

> 心态层面：「事后被证伪」≠「当时判断错误」。负面知识（此路不通 + 原因是 X）本身就是产出。
